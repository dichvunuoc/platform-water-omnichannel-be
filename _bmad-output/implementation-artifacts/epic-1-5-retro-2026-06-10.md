## What Went Well

### 1. 🏆 Canonical BFF Module Pattern (Story 2.1)
Story 2.1 (Customer Profile 360°) defined the pattern that every subsequent story copied:
`Controller → CQRS → Handler → PortRegistry → Adapter → Downstream`
After Story 2.1, velocity increased because there was zero architectural debate.

### 2. 🏆 SessionAuthGuard + @CurrentUser() Decorator (R-1)
Eliminated `getAuthenticatedUserId()` boilerplate across ALL controllers. Global APP_GUARD + `@Public()` decorator + `@CurrentUser('id')` param decorator. 53 new tests, zero regressions.

### 3. 🏆 740+ Tests, Zero Regressions
Test count grew organically from 30 to 740+ across 19 stories. Every story added meaningful tests covering port adapters, handlers, controllers, and integration flows.

### 4. 🏆 Code Review Discipline — 60+ Findings Caught
12 formal code reviews caught critical issues: H1 global guard blocking login (R-1), wrong exception types (5.1, 5.2), missing idempotency (5.2), import path errors (5.3), stale mock data (3.3).

### 5. 🏆 Reusable Webhook Infrastructure (Story 4.2)
Created three pieces of shared infrastructure: `InterServiceApiKeyGuard`, `deleteByPattern()` on CacheService, and the webhook controller pattern. Story 5.2 reused all three.

### 6. 🏆 BFF Pass-Through Principle Upheld
Across all 19 stories, zero business logic leaked into BFF. Only presentation transformations: `percentageChange` (3.1), `isWarning` (2.3), aging buckets (4.5).

### 7. 🏆 Pattern Convergence Across Epics
By Epic 3, every story followed identical patterns: `useExisting` for DI, `onModuleInit` for port registration, `MockAdapterBase` extension, Zod schema maps, integration tests.

## What Didn't Go Well — Challenges

### 1. ⚠️ `getAuthenticatedUserId()` Duplication (Stories 2.1–2.3)
Auth extraction method copy-pasted across 3 controllers before R-1 refactoring addressed it. Lesson: flag duplication after 2nd occurrence, don't wait.

### 2. ⚠️ Import Path Errors Masked by `import type` (Story 5.3)
`import type` with wrong relative paths — TypeScript erases type-only imports at compile time, so errors didn't surface until runtime.

### 3. ⚠️ Zod v4 Breaking Changes (Stories 1.1, 2.1)
Architecture doc referenced Zod v3 patterns but project uses Zod v4: `z.record()` needs 2 args, `z.string().email()` → `z.email()`.

### 4. ⚠️ Wrong Exception Types (Stories 5.1, 5.2)
`NotFoundException` (HTTP 404) used for downstream failures instead of `PortFallbackException` (HTTP 500). The resource exists; the service just couldn't reach it.

### 5. ⚠️ Pre-existing Infrastructure Assumptions (Story 1.1)
Architecture doc said to install `opossum` for Circuit Breaker. Codebase already had fully implemented custom `CircuitBreakerState`. Almost installed redundant library.

### 6. ⚠️ Story 4-5 Stuck in Review
574 tests passing, zero regressions, but code review sign-off not completed. Process issue — review shouldn't be a bottleneck.

### 7. ⚠️ Story 1.2 Status Discrepancy
Sprint-status shows "done" but story file shows "ready-for-dev". ACs were functionally delivered across Stories 1.1, 1.4, and 4.2 — tracking artifact, not a real gap.

### Team Agreements

1. Always use `PortFallbackException` (HTTP 500) for downstream failures — never `NotFoundException` (HTTP 404)
2. Pass `validated.error.message` to `ValidationException` — field-level error info, not generic strings
3. Extract magic strings as named constants
4. Static NestJS routes before dynamic params (`/search` before `/:articleId`)
5. `useExisting` (not `useClass`) for DI token providers
